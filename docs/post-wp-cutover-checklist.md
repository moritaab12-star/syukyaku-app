# WordPress 廃止後の本番チェックリスト

Next 単体運用へ切り替えた**あと**に、インデックス・ブランド・トラフィックを壊さないための順序です。依存関係を崩さないよう、次の流れを推奨します。

## 運用上の安全な順序（概要）

1. **本番 Next を最新にデプロイ**（公開フロー・LP は既に Next のみであることを確認）
2. **`vercel.json` の `redirects` を埋めて再デプロイ**  
   - **DNS が既に Vercel を向いている**場合: 先に 301 を入れておくと、旧ブックマークが即 `/p/{slug}` へ流れる  
   - **いまドメインが旧 WP を向いている**場合: DNS 切替と同日（または直後）に、**Vercel 側のプロジェクトに同じホスト名**が付いた状態でリダイレクトを有効化する。表で `OLD_PATH` → `/p/{slug}` を一覧化し、`docs/vercel-redirects.template.json` を参考に `vercel.json` の `redirects` に追記する
3. **DNS を Vercel（または合意したホスト）へ切替**（未切替ならこのタイミング）
4. **本番で 301 を実 URL で確認**（主要な旧 LP パスから Next の LP へ一本化されているか）
5. **Search Console**  
   - ドメインまたは URL プレフィックスの**プロパティ**が変わる場合は追加・所有権確認  
   - **適用されたら**「インデックス登録」→**サイトマップ**を再送（例: `https://{本番ドメイン}/sitemap.xml`）
6. **本番環境変数から `WORDPRESS_*` を削除**（コードは参照しない想定。残すと設定ミスの温床になる）
7. **旧 WordPress サーバの停止・契約終了**（トラフィックと DNS が Next 側で安定してから。早めると 301 や検証ができない）

## DNS 切替

- [ ] 本番ドメインの A / CNAME が意図したホスト（Vercel 等）のみを指している
- [ ] TTL を把握し、切替後の伝播待ち時間を確保した

## 301（リダイレクト）

- [ ] 旧 WP のパス一覧が表で整備されている（`OLD_PATH` → `NEXT_SLUG`）
- [ ] `vercel.json` の `redirects` に反映済み（テンプレは `docs/vercel-redirects.template.json`）
- [ ] ブラウザで主要 URL を開き、`/p/{slug}` へ恒久的に遷移することを確認した

## Search Console

- [ ] 本番 URL に対応するプロパティで所有権済み
- [ ] サイトマップ URL を送信した（または再送した）

## サイトマップ

- [ ] `robots.txt` が本番で期待どおり（サイトマップ URL を含む）
- [ ] `/sitemap.xml` が 200 で、公開中の LP が含まれる

## 環境変数（`WORDPRESS_*` 等）

- [ ] Vercel（本番）の Environment Variables から `WORDPRESS_*` を削除した
- [ ] ローカル `.env.local` にも本番用の WP 認証が残っていない（秘密の取り扱いに注意）

## 旧 WP サーバ

- [ ] ブックマーク・外部リンクが Next へ誘導されることを確認したうえで、旧サーバを停止またはリダイレクト専用から切り離した

---

## F1: マージ後に人がやることだけ（チェックリスト）

コード・設定ファイルのマージが完了した**あと**、以下は人手で順に実施する。

- [ ] 旧 WP パス → Next slug の対応表を確定させた
- [ ] 対応表に基づき `vercel.json` の `redirects` を埋め、本番へデプロイした
- [ ] DNS を本番 Next 側に向けた（必要な場合）
- [ ] 本番で 301 と LP 表示を実 URL 検証した
- [ ] Search Console でサイトマップを送った（または再送した）
- [ ] 本番ホストから `WORDPRESS_*` 環境変数を削除した
- [ ] 旧 WordPress を停止・解約した（トラフィック確認後）
